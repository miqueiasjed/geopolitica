# Task 10.1 – Migrations e Models do Sistema de Alertas

## Objetivo
Criar as 3 migrations (`sinais_padrao`, `alertas_preditivos`, `alertas_leituras`) e os Models correspondentes com relacionamentos e scopes.

## Tipo
backend-estrutura

## Arquivos a criar/modificar
- `database/migrations/xxxx_create_sinais_padrao_table.php` → criar
- `database/migrations/xxxx_create_alertas_preditivos_table.php` → criar
- `database/migrations/xxxx_create_alertas_leituras_table.php` → criar
- `app/Models/SinalPadrao.php` → criar
- `app/Models/AlertaPreditivo.php` → criar
- `app/Models/AlertaLeitura.php` → criar

- laravel-arquitetura

## Especificação

### O que fazer:

1. **Migration `sinais_padrao`**:
   - `id`
   - `event_id` (foreignId, constrained `events`, onDelete cascade)
   - `tipo_padrao` (string — `military` | `diplomatic` | `supply`)
   - `nome_sinal` (string — ex: "Mobilização Militar", "Ruptura de Fornecimento")
   - `regiao` (string)
   - `peso` (tinyInteger, 1–10)
   - `confianca` (decimal 5,2 — porcentagem de confiança da IA, 0–100)
   - `analisado_em` (timestamp)
   - `timestamps`
   - Índice em `(event_id, tipo_padrao)` e `(regiao, analisado_em)`

2. **Migration `alertas_preditivos`**:
   - `id`
   - `nivel` (string — `medium` | `high` | `critical`)
   - `regiao` (string)
   - `titulo` (string)
   - `analise` (text — análise IA gerada)
   - `resumo_sinais` (json — array com título+tipo de cada sinal)
   - `peso_total` (integer)
   - `tipos_padrao` (json — array de tipos envolvidos: `['military', 'supply']`)
   - `notificado_em` (timestamp, nullable — quando e-mails foram enviados)
   - `timestamps`
   - Índice em `(regiao, created_at)`, `nivel`, `notificado_em`

3. **Migration `alertas_leituras`**:
   - `id`
   - `user_id` (foreignId, constrained `users`, onDelete cascade)
   - `alerta_id` (foreignId, constrained `alertas_preditivos`, onDelete cascade)
   - `lido_em` (timestamp)
   - `timestamps`
   - UNIQUE constraint em `(user_id, alerta_id)`

4. **Model `SinalPadrao`**:
   - `$fillable` com todos os campos
   - Cast `confianca` para float, `analisado_em` para datetime
   - Relacionamento `evento()`: belongsTo `Event`
   - Scope `recentes()`: `where('analisado_em', '>=', now()->subHours(48))`
   - Scope `porRegiao(string $regiao)`

5. **Model `AlertaPreditivo`**:
   - `$fillable` com todos os campos
   - Casts: `resumo_sinais` e `tipos_padrao` → array, `notificado_em` → datetime
   - Relacionamento `leituras()`: hasMany `AlertaLeitura`
   - Scope `naoLidosPor(int $userId)`: alertas sem registro em `alertas_leituras` para o usuário
   - Scope `visivelPara(string $role)`: filtra por nível conforme hierarquia de plano

6. **Model `AlertaLeitura`**:
   - `$fillable`: `user_id`, `alerta_id`, `lido_em`
   - Cast `lido_em` para datetime
   - Relacionamentos: belongsTo `AlertaPreditivo` e `User`

### Regras de negócio aplicáveis:
- `(user_id, alerta_id)` UNIQUE em `alertas_leituras` — impede duplo "lido"
- Scope `visivelPara`: `essencial` → apenas `medium`; `pro` → `medium`+`high`; `reservado`/`admin` → todos
- `notificado_em` nulo significa que o e-mail ainda não foi enviado

### Critérios de aceitação:
- [ ] `php artisan migrate` cria as 3 tabelas sem erros
- [ ] UNIQUE constraint em `alertas_leituras(user_id, alerta_id)` funciona
- [ ] `AlertaPreditivo::visivelPara('assinante_essencial')` retorna apenas nível `medium`
- [ ] `AlertaPreditivo::naoLidosPor($userId)` exclui alertas já lidos
