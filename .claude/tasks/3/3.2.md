# Task 3.2 – HotmartHandlerService: handlers para cada tipo de evento

## Objetivo
Implementar o `HotmartHandlerService` com handlers para os 7 tipos de eventos da Hotmart, incluindo criação automática de User + Assinante, atribuição de roles e atualização de status.

## Tipo
backend-logica

## Arquivos a criar/modificar
- `app/Services/HotmartHandlerService.php` → criar
- `app/Mail/BoasVindasMail.php` → criar (Mailable simples)

## Contexto necessário (arquivos para ler antes)
- `app/Models/User.php` → ver campos e relacionamentos
- `app/Models/Assinante.php` → ver campos `plano`, `ativo`, `status`
- `database/seeders/RolesSeeder.php` → confirmar nomes exatos dos roles

## Skills necessárias
- laravel-arquitetura
- permissoes-e-sync

## Especificação

### O que fazer:
1. Criar `HotmartHandlerService` com método público `handle(WebhookEvento $evento): void`
   - Roteia para handler privado baseado em `$evento->event_type`
2. Implementar handlers privados:
   - `handlePurchaseApproved(array $payload)`:
     - Criar ou buscar `User` pelo email (`firstOrCreate`)
     - Criar `Assinante` com `plano` derivado do produto Hotmart, `ativo = true`, `status = ativo`
     - Atribuir role Spatie baseado no plano (`assinante_essencial`, `assinante_pro`, `assinante_reservado`)
     - Disparar `BoasVindasMail` com link de reset de senha
   - `handlePurchaseComplete(array $payload)`: idêntico a `Approved` (idempotente)
   - `handlePurchaseCanceled(array $payload)`: `ativo = false`, `status = cancelado`
   - `handlePurchaseRefunded(array $payload)`: `ativo = false`, `status = reembolsado`
   - `handlePurchaseChargeback(array $payload)`: `ativo = false`, `status = chargeback`
   - `handlePurchaseExpired(array $payload)`: `ativo = false`, `status = expirado`
   - `handleSwitchPlan(array $payload)`: atualizar `plano` e role Spatie, manter `ativo = true`
3. Criar `BoasVindasMail`:
   - Mailable com `$user` e link de reset de senha
   - Usar `Password::createToken($user)` para gerar o link
   - Link apontando para `{FRONTEND_URL}/redefinir-senha?token={token}&email={email}`

### Regras de negócio aplicáveis:
- `PURCHASE_APPROVED` e `PURCHASE_COMPLETE` devem ser idempotentes (usar `firstOrCreate`)
- Ao trocar plano (`SWITCH_PLAN`): remover role anterior antes de adicionar novo
- O service não deve lançar exceptions que parem a execução — capturar e registrar em `WebhookEvento.erro`

### Critérios de aceitação:
- [ ] `PURCHASE_APPROVED` cria User + Assinante + role + envia e-mail
- [ ] Dois eventos `PURCHASE_APPROVED` para o mesmo email não duplicam usuário
- [ ] `PURCHASE_CANCELED` define `ativo = false` e `status = cancelado`
- [ ] `SWITCH_PLAN` atualiza plano e substitui role Spatie
- [ ] `BoasVindasMail` gera link correto com token de reset
