# Task 31.6 – Refactor AdminAssinanteAddonController + FormRequests

## Objetivo
Atualizar `AdminAssinanteAddonController` para atribuir `assinante` ao criar usuário durante importação de addons (linhas ~216 e ~273), e atualizar as FormRequests para aceitar `assinante` como role válida de assinante.

## Tipo
backend-endpoint

## Arquivos a criar/modificar
- `app/Http/Controllers/Api/Admin/AdminAssinanteAddonController.php` → modificar
- `app/Http/Requests/CriarUsuarioRequest.php` → modificar
- `app/Http/Requests/AtualizarUsuarioRequest.php` → modificar

## Contexto necessário (arquivos para ler antes)
- `app/Http/Controllers/Api/Admin/AdminAssinanteAddonController.php` → localizar linhas ~216 e ~273
- `app/Http/Requests/CriarUsuarioRequest.php` → ver regra de validação para o campo `role`
- `app/Http/Requests/AtualizarUsuarioRequest.php` → idem

## Skills necessárias
- `laravel-arquitetura`

## Especificação

### O que fazer:
**AdminAssinanteAddonController:**
1. Linha ~216: substituir atribuição de role específica (ex: `$usuario->assignRole('assinante_'.$plano)`) por:
   ```php
   $usuario->syncRoles(['assinante']);
   ```
2. Linha ~273: idem — mesma substituição para o segundo ponto de criação de usuário.
3. Verificar se há outros pontos no arquivo que atribuem roles de assinante específicas.

**CriarUsuarioRequest e AtualizarUsuarioRequest:**
1. Localizar a regra de validação do campo `role` (ex: `'role' => 'in:assinante_essencial,assinante_pro,assinante_reservado,admin,...'`).
2. Substituir as três roles específicas por `assinante`:
   - `in:assinante,admin,editor,...` (manter outras roles não relacionadas a assinante).
3. Garantir que a validação aceita `assinante` sem aceitar as roles antigas.

### Regras de negócio aplicáveis:
- O controller cria novos usuários durante importação de addons — eles SEMPRE devem receber role `assinante`.
- As FormRequests definem as roles válidas para operações manuais do admin — `assinante` é a única role de assinante válida após este plano.
- Não remover outras roles não relacionadas a assinante (ex: `admin`, `editor`).

### Critérios de aceitação:
- [ ] Linhas ~216 e ~273 de `AdminAssinanteAddonController` atribuem `assinante` (não roles específicas).
- [ ] `CriarUsuarioRequest` aceita `role = 'assinante'` e rejeita `role = 'assinante_pro'`.
- [ ] `AtualizarUsuarioRequest` idem.
- [ ] Nenhuma referência a `assinante_essencial/pro/reservado` permanece nos 3 arquivos.

## Teste esperado
