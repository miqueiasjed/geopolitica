# Task 27.2 – FormRequests AdicionarAddonRequest e AtualizarAddonRequest

## Objetivo
Criar os dois FormRequests para gestão manual de addons por assinante, validando `addon_key` contra a tabela `produtos` e `status`/`fonte` contra valores permitidos.

## Tipo
backend-logica

## Arquivos a criar/modificar
- `backend/app/Http/Requests/Admin/AdicionarAddonRequest.php` → criar
- `backend/app/Http/Requests/Admin/AtualizarAddonRequest.php` → criar

## Contexto necessário (arquivos para ler antes)
- `backend/app/Models/AssinanteAddon.php` → ver constantes FONTES, STATUS
- `backend/app/Http/Requests/Admin/CriarProdutoRequest.php` → ver padrão de authorize() admin
- `backend/app/Models/Produto.php` → confirmar coluna `chave` para Rule::exists

## Skills necessárias
- `laravel-arquitetura`

## Especificação

### O que fazer:
**AdicionarAddonRequest:**
```php
public function rules(): array
{
    return [
        'addon_key'   => ['required', 'string', Rule::exists('produtos', 'chave')],
        'status'      => ['required', Rule::in(['ativo', 'cancelado', 'expirado', 'reembolsado'])],
        'fonte'       => ['required', Rule::in(['hotmart', 'lastlink', 'manual'])],
        'iniciado_em' => ['nullable', 'date'],
        'expira_em'   => ['nullable', 'date', 'after_or_equal:iniciado_em'],
    ];
}
```

**AtualizarAddonRequest:**
- Mesmas regras mas com `sometimes` em todos os campos
- `addon_key` read-only: não aceitar alteração (omitir da validação)
- Não requer `fonte`

### Regras de negócio aplicáveis:
- `addon_key` deve existir na tabela `produtos` (Rule::exists)
- `fonte` só pode ser `hotmart`, `lastlink` ou `manual`
- `expira_em` deve ser posterior ou igual a `iniciado_em` quando ambos fornecidos

### Critérios de aceitação:
- [ ] `addon_key` inexistente em `produtos` retorna 422
- [ ] `status` inválido retorna 422
- [ ] `fonte` diferente de `hotmart|lastlink|manual` retorna 422
- [ ] `expira_em` anterior a `iniciado_em` retorna 422
- [ ] Usuário não-admin recebe 403
