# Task 27.4 – AdminAssinanteAddonController (PUT editar + DELETE remover)

## Objetivo
Adicionar os métodos `update` e `destroy` no `AdminAssinanteAddonController`, mantendo `assinantes.addons[]` sincronizado via `AddonService` após mudanças de status.

## Tipo
backend-endpoint

## Arquivos a criar/modificar
- `backend/app/Http/Controllers/Api/Admin/AdminAssinanteAddonController.php` → modificar

## Contexto necessário (arquivos para ler antes)
- `backend/app/Http/Controllers/Api/Admin/AdminAssinanteAddonController.php` → estado após Task 27.3
- `backend/app/Services/AddonService.php` → entender métodos `ativar()` e `cancelar()`
- `backend/app/Models/AssinanteAddon.php` → ver constantes STATUS

## Skills necessárias
- `laravel-arquitetura`

## Especificação

### O que fazer:
**update(AtualizarAddonRequest $request, User $user, AssinanteAddon $addon):**
- Atualizar o addon com `$request->validated()`
- Se o novo status for `ativo` → chamar `AddonService::ativar($user, $addon->addon_key)`
- Se o novo status for `cancelado|expirado|reembolsado` → chamar `AddonService::cancelar($user, $addon->addon_key)` (ou equivalente)
- Retornar addon atualizado com 200

**destroy(User $user, AssinanteAddon $addon):**
- Verificar que `$addon->user_id === $user->id` (404 se não pertencer)
- Deletar o registro
- Retornar 204

### Regras de negócio aplicáveis:
- Sincronizar `assinantes.addons[]` sempre que o status mudar
- Ao deletar, não sincronizar automaticamente (remoção direta do registro)
- Validar que o addon pertence ao assinante informado na URL

### Critérios de aceitação:
- [ ] `PUT` com `status=ativo` sincroniza addon no assinante
- [ ] `PUT` com `status=cancelado` remove addon do `assinantes.addons[]`
- [ ] `DELETE` remove registro e retorna 204
- [ ] `DELETE` de addon de outro assinante retorna 404
- [ ] Usuário não-admin recebe 403
