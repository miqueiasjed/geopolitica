# Plano 3 – Integração Hotmart (M10)

## Objetivo
Automatizar o ciclo de vida do assinante conectando a Hotmart ao Laravel via webhook — da compra ao cancelamento — eliminando intervenção manual do administrador.

## Contexto de Negócio
Toda conta de assinante é criada automaticamente quando a Hotmart dispara um evento de compra aprovada. O sistema processa 7 tipos de eventos: `PURCHASE_APPROVED`, `PURCHASE_COMPLETE`, `PURCHASE_CANCELED`, `PURCHASE_REFUNDED`, `PURCHASE_CHARGEBACK`, `PURCHASE_EXPIRED` e `SWITCH_PLAN`. O endpoint de webhook **sempre** retorna HTTP 200 (evita reenvios da Hotmart). Segurança via header `x-hotmart-webhook-token`.

Detalhes completos em: `prd/modulo-10-hotmart-pagamentos.md`

## Escopo

### Inclui
- Migration `webhook_eventos` (log de todos os eventos recebidos da Hotmart)
- `WebhookHotmartController` — recebe, valida e roteia eventos
- `HotmartHandlerService` — handlers para cada tipo de evento (criar user, assignRole, ativar/desativar, trocar plano)
- Criação automática de `User` + `Assinante` + role na compra aprovada
- Disparo de e-mail de boas-vindas com link de definição de senha (Laravel Password Reset)
- 4 templates Blade de e-mail: boas-vindas, acesso liberado, cancelamento, reembolso
- Painel admin React: `AdminAssinantes.tsx` (listagem com busca), `AdminWebhookEventos.tsx` (log de eventos)
- Rota pública `/api/webhook/hotmart` (sem auth, com validação de token)
- Rota admin `/api/admin/assinantes` (listagem, filtro por plano/status)

### Não inclui (fica para outro plano)
- Dashboard do assinante → já coberto pelos módulos de conteúdo

## Dependências

## Entidades/Models envolvidos
- `User` — criado automaticamente na compra
- `Assinante` — atualizado em cada evento (plano, status, ativo)
- `WebhookEvento` — log de todos os eventos recebidos (`event_type`, `payload`, `processado_em`)
