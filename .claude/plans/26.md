# Plano 26 – Multi-Produtos: Backend Core (Catálogo + Endpoint de Status + CRUD Admin)

## Objetivo
Criar o catálogo de produtos configurável no banco de dados, expor o endpoint `/api/meus-produtos` com status por usuário, e entregar o CRUD admin de produtos.

## Contexto de Negócio
O sistema atual trata extras ("Monitor de Guerra", "Monitor Eleitoral") como addons com chaves fixas e links hardcoded em env vars — impossível alterar sem deploy. Este plano cria a tabela `produtos` para armazenar essas configurações de forma dinâmica, expõe o status do addon por usuário autenticado e permite ao admin gerenciar os produtos pelo painel.

Detalhes completos em: `/Users/miqueias/.claude/plans/alguns-assinantes-tem-planos-swirling-lamport.md`

## Escopo

### Inclui
- Migration `produtos` (chave, nome, descricao, preco_label, link_compra, link_reativar, ativo, ordem)
- Model `Produto` com `$fillable`, casts, `scopeAtivo()`, `scopeOrdenado()`
- Seeder `ProdutoSeeder` com `updateOrCreate` para `elections` e `war` lendo URLs das env vars; registro em `DatabaseSeeder`
- Controller `MeusProdutosController` — `GET /api/meus-produtos` (auth:sanctum) com lógica de status por usuário (admin → ativo; `temAddon` → ativo; último registro de `AssinanteAddon` → retorna status; sem registro → null)
- Controller `AdminProdutoController` — `index`, `store`, `update`, `destroy` (destroy bloqueia com 422 se houver addons ativos)
- FormRequests `CriarProdutoRequest` e `AtualizarProdutoRequest` com validação de `chave` (regex `^[a-z0-9_-]+$`, unique no create, read-only no update)
- Rotas em `routes/api.php` (grupo auth:sanctum e grupo admin)
- Adicionar `'manual'` à constante `AssinanteAddon::FONTES`

### Não inclui (fica para outro plano)
- Gestão manual de addons por assinante → Plano 27
- Importação/exportação em lote → Plano 27
- Frontend → Planos 28 e 29
