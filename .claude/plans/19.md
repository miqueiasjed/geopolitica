# Plano 19 – Add-ons: Banco de Dados e Serviço Central

## Objetivo
Criar a infraestrutura de dados e a lógica central de add-ons, incluindo migrations, o `AddonService` compartilhado, e a integração com o `HotmartHandlerService` existente para suportar compra de add-ons via Hotmart.

## Contexto de Negócio
O sistema ganha uma segunda camada de monetização além dos planos (Essencial, Pro, Reservado): **add-ons comprados avulsamente**. Há dois add-ons:
- `elections` — Monitor Eleitoral — R$ 297/ano (disponível para Essencial)
- `war` — Monitor de Guerra — R$ 497/ano (disponível para Essencial e Pro)

Regras de acesso combinadas:
- **Essencial** → GPI Core. Add-ons desbloqueiam verticais específicas.
- **Pro** → Core + Monitor Eleitoral incluído (sem precisar do add-on `elections`).
- **Reservado** → Core + Monitor Eleitoral + Monitor de Guerra incluídos.

O `AddonService` é o único ponto de ativação/cancelamento — chamado tanto pelo webhook Hotmart (plano 19) quanto pelo webhook Lastlink (plano 20).

## Escopo

### Inclui
- Migration: coluna `addons` (JSON, default `[]`) na tabela `assinantes`
- Migration: tabela `assinante_addons` (histórico de add-ons: user_id, addon_key, status, lastlink_order_id, hotmart_order_id, started_at, expires_at)
- Migration: coluna `vertical_conteudo` (nullable enum: `elections`, `war`) na tabela `conteudos`
- `AddonService` com métodos `ativar(userId, addonKey, orderId, fonte)` e `cancelar(userId, addonKey, motivo)`
- Atualizar `HotmartHandlerService`: detectar se produto é add-on (via config) e delegar ao `AddonService`
- Atualizar middleware/gate de autorização para verificar `addons` além de `plano`
- Seeder de produtos Hotmart para add-ons (configuração de IDs)

### Não inclui
- Webhook Lastlink → Plano 20
- Páginas frontend das verticais → Plano 21
- E-mail de boas-vindas do add-on → Plano 20

## Dependências
